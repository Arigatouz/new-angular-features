Why userForm().pending()
never saw the phone field

A field's pending() is true while an async validator resolves. The surprise is how far up the form tree that truth travels.

TL;DR. pending() bubbles up exactly one level. Your async validator sits on phone, which is a grandchild of the root, so userForm().pending() stays false the whole time. Its sibling states (valid, invalid, touched, dirty) recurse the full tree. pending is the odd one out.

1 The setup

An async validator on the phone field. It debounces, then simulates a "is this number already registered" check.

validateAsync(path.personalInfo.phone, {
  params: ({ value }) => value(),
  when:   ({ value }) => value().trim().length > 0,
  debounce: 500,
  factory: (params) => resource({
    params: () => params(),
    loader: async ({ params }) => {
      await new Promise((r) => setTimeout(r, 1500));   // fake network
      return REGISTERED_PHONE_NUMBERS.includes(params);
    },
  }),
  onSuccess: (taken) => taken ? { kind: 'phoneTaken', message: '...' } : null,
  onError:   () => ({ kind: 'phoneCheckFailed', message: '...' }),
});

Timeline: 500ms debounce + 1500ms loader = about 2 seconds where phone().pending() is true.

3 pending vs everything else

Every aggregated state uses the same reduceChildren walk over direct children. The difference is what each one asks the child.

Recurses the whole tree ✓

Reads the child's own aggregated signal, so each child re-runs the same walk on its children. Truth flows all the way to the root.

validinvalid toucheddirty errorSummary 
// dirty — calls child.dirty() → recursive
value || child.nodeState.dirty()
Only one level ✗

Reads the child's own raw async errors, not the child's aggregated pending. So a grandchild's pending never reaches the top.

pending 
// pending — reads raw errors, NOT recursive
value || child.validationState
  .asyncErrors().includes('pending')

Straight from the compiled source. Note line 751 reads asyncErrors(), while dirty at line 1643 reads child.nodeState.dirty(). That single choice is the entire bug.

5 The practical fix

Your submit guard on the root won't block anything while the phone validates. Guard on the field (or its immediate group) instead.

✗ does nothing   [disabled]="userForm().submitting() || userForm().pending()"
✓ actually blocks   [disabled]="userForm().submitting() || userForm.personalInfo.phone().pending()"

Had the async validator been on a direct child of the root, userForm().pending() would have caught it. It is specifically the two-levels-deep case that slips through. This reads as an inconsistency (very likely a bug) in this nightly build, since pending is the only aggregated state that does not recurse.
